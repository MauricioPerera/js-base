## Resumen ejecutivo

Lo que empezó como "analizá el proyecto js-store" derivó en: la evolución de **js-store** por
cinco releases, la construcción desde cero del MVP **js-base** (backend estilo PocketBase con
búsqueda semántica), el cierre completo de una **auditoría externa** de cinco hallazgos, y una
**optimización de rendimiento** diagnosticada y medida. Los tres repos de la cadena quedan
públicos, con CI verde y metadata coherente.

## js-base v0.1.3 — el MVP

Backend embebido estilo PocketBase con búsqueda semántica nativa, JavaScript puro, cero
dependencias. Construido en **7 batches KDD** (contratos + tests congelados):

- **B0** scaffold + vendoring congelado por hash · **B1** collections + auth-service + adapter
  atómico · **B2** núcleo HTTP (router, hooks, pipeline) · **B3** REST (records CRUD / auth /
  files) · **B4** rules engine (`matchFilter`) + realtime SSE · **B5** endpoints semánticos +
  harness adversarial · **B6** bootstrap `createServer` + CLI.
- **Diferenciador:** búsqueda vectorial e híbrida (vector + BM25) por HTTP, verificada e2e.
- **Cierre adversarial:** harness con crash-injection SIGKILL + fuzz de auth/rules que **cazó un
  bug real de durabilidad** — el sistema funcionando como se diseñó.
- **Publicado:** repo público, 14 topics, descripción, LICENSE, badge CI, GIF de demo, ejemplos de
  uso verificados contra server real, homepage a la sección de ejemplos.

## Auditoría externa — cerrada entera

| # | Hallazgo | Estado |
|---|---|---|
| 1 | `_files` escritura/borrado anónimo | ✅ Fix: exige auth (js-base v0.1.1) |
| 2 | Sin rate limiting en `/login` | ✅ Documentado en Límites conocidos |
| 3 | `_sessions` crece sin límite | ✅ Documentado (confirmado en el vendor: verify stateless, sin purga) |
| 4 | Compare de hash no timing-safe | ✅ Fix en cascada 3 repos (js-doc-store → js-store → js-base) |
| 5 | searchHybrid/reindex no medidos | ✅ Benchmark + hallazgo derivado (bulk-load O(N²)) fixeado |

**El #5 destapó el bulk-load O(N²):** se diagnosticó descartando fsync con un experimento de
aislamiento (misma degradación con fsync anulado → no era I/O), se encontró la causa real
(`upsert` → `remove({_id})` escaneaba todo el log) y se arregló con un fast-path de clave
primaria. Medido: **60 → 674 docs/s a 10k, y ya plano** (O(N)).

## Verificación (por artefacto, nunca por la palabra del dev)

Sondas adversariales propias corridas antes de integrar cada pieza:

- Path-traversal contra `/api/files` (`%2F` codificado → 403, cero escape de directorio).
- `{"auth.id":{$ne:null}}` fallaba **abierto** para "requiere login" → convención corregida a `$exists`.
- Búsqueda semántica end-to-end sobre HTTP (rankeo por coseno).
- Registro torn en `DiskKV` (crash reproducido antes de arreglarlo).
- Conteo de `_scan` (500 upserts = 500 scans → confirmó el O(N²)).
- Fix de files auth, timing-safe, y el diff de cada re-vendor = exactamente el fix (sin drift).
